Progress

Où en est-on.

Festiflow est né d'outils isolés construits pour Madame Loyal. L'objectif : une plateforme unifiée, d'abord en interne, puis en SaaS pour l'industrie événementielle.

Origines

Le projet a démarré par des outils standalone

Festiflow est né du besoin concret de remplacer les fichiers Excel et Google Sheets utilisés pour gérer les opérations de Madame Loyal Festivals. Les premiers outils — Billetterie, Budget et Médias (Dropbox Manager) — ont été construits en HTML standalone pour répondre aux urgences opérationnelles. La Billetterie a prouvé le concept : import CSV manuel des données DICE et Shotgun, courbes de ventes cumulatives, projections multi-scénarios — le tout opérationnel et utilisé en production. Le succès de ces standalones a validé la décision de construire une plateforme complète.

Phase actuelle

Design React — module par module

Chaque module est designé en React avec le design system Festiflow avant d'être migré sur le serveur. Le shell (navigation, structure, composants de base) est en cours de finalisation. Le Budget sera le premier module à passer en design React complet, suivi des Contacts. Les modules restants seront designés progressivement — certains migreront vers le serveur en parallèle dès que l'infrastructure sera prête.

Cette approche permet au CEO de valider chaque module visuellement et fonctionnellement avant le développement serveur — pas de surprises à la livraison.

Migration serveur

C'est là que tout prend vie

La phase design produit l'interface — la migration serveur est là où le vrai travail d'ingénierie commence. Chaque module designé doit être connecté à une infrastructure réelle : création des bases de données, définition des schémas, construction des APIs, connexion entre modules. Un contact saisi dans le Registre doit apparaître dans le Catering, le Transport et l'Hébergement — ça implique des relations de données, des endpoints partagés et une logique métier qui relie tout.

Le travail se fait module par module, étape par étape. Le Budget sera le premier à être opérationnel sur le serveur — base de données, API, import Qonto automatisé, données réelles. Les Contacts suivront comme fondation partagée. Puis chaque nouveau module se branche sur l'existant, partageant les mêmes bases de données et communiquant via des APIs internes. Les standalones HTML actuels seront progressivement remplacés par leurs versions React connectées.
